# Obsidian Vault — Claude Code 全局指令

这是用户的 Obsidian 知识库，作为 Claude Code 的"项目上下文缓存"。

## 核心原则

**先读 Obsidian，再读代码**——但不要盲信 Obsidian。把 Obsidian 当地图，代码当地形：地图建立心智模型，地形决定动手前的真相。

## 目录结构

| 文件夹 | 用途 | 写入规则 |
|--------|------|---------|
| `项目/` | 每个代码仓库一个子文件夹 | 通过 `/add-project` 创建模板，手动编辑详情 |
| `项目/<name>/概览.md` | 项目入口 | 架构有重大变化时更新 |
| `项目/<name>/activeContext.md` | **每次开工必读** | 切换焦点时手动更新；最近改动由 `/sync-docs` 自动维护 |
| `项目/<name>/progress.md` | 已完成功能 + 决策记录 | 完成里程碑或拍决策时手动追加 |

## 查阅优先级 + 触发规则清单

### 必须先读 Obsidian 的场景

- 用户问"X 项目/系统是怎么 Y 的"
- 用户说"加 / 改 / 重构 X 功能" → 动手前先读 `项目/<X>/02-功能/`
- 跨仓库任务 → 先读 `首页.md` 的依赖图
- 用户提"上次 / 之前 / 那个决策" → 看对应 `progress.md`

### 可以跳过的场景

- 纯代码改动（typo / rename / 加日志）
- 用户给了完整上下文，没让你"理解"什么
- 只读查询（git log / grep / cat）

## Obsidian vs 代码的分工（地图 vs 地形）

| 信息类型 | 真相源 | Obsidian 角色 |
|---------|--------|---------------|
| 代码现状（函数签名、调用链、实际行为） | 代码 | 缓存/索引，可能过期 |
| 设计意图 / 历史决策 / tradeoff | Obsidian | 唯一真相 |
| 当前焦点 / 进度 | Obsidian | 唯一真相 |
| 跨仓库依赖关系 | 两边核对 | 起点（声明），代码确认（实际 import） |

**改代码前必须 Read 实际文件**，不能凭 Obsidian 描述就改。

发现不一致时：小偏差立刻修 Obsidian / 大偏差停下来问用户 / **不要假设哪边对**。

## 写入约定

- 跨文件链接用全路径 `[[项目/<name>/概览]]`
- 修改代码后用 `/sync-docs` 同步对应项目文档
- 项目文档入口统一命名 `概览.md`，不要用裸数字前缀
- 标记"此区域由 X 自动维护"的部分可以覆写，其余不要碰

## Git 操作约定（重要）

**用户提到 commit / push / pull / 提交 / 推送 / 拉取时**：

- **优先用 `/commit` 和 `/pull`**，不要直接跑裸 `git commit` `git pull`
- 这两个 slash command 会自动同步 vault 文档（activeContext / 首页状态），跳过它们会让 Obsidian 文档和实际代码状态漂移
- 仅当以下场景才用裸 git：
  - 用户**明确说**"裸 git" / "不要走 skill" / "用原始 git"
  - 操作复杂到 skill 不支持（rebase、cherry-pick、reflog、bisect 等）
  - 用户在 review 阶段问"显示 diff"等只读查询

**裸 git 操作完成后**，主动建议用户跑一次 `/sync-docs` 补同步：

> "已经用裸 git 提交。要不要跑 `/sync-docs` 同步到 Obsidian？"

**为什么这条特别重要**：本 vault 的核心价值是"Claude 下次开工前先读 Obsidian 拿到上下文"。如果文档和代码漂移，下次会话里 Claude 看到的项目状态是过期的，所有判断都会偏。
