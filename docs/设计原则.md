# 设计原则

## 1. 文档是上下文缓存，不是文档

我们不是为了"写好文档"而写文档，而是为了**让 Claude 每次开工前能 5 秒进入状态**。

衡量标准：Claude 读完 `概览.md + activeContext.md` 后，能不能直接干活？如果不行，文档就该改。

## 2. 区分静态和动态

| 类型 | 文件 | 更新频率 |
|------|------|---------|
| 静态（架构、产品） | `概览.md`、`01-架构/`、`03-产品/` | 重大变化时 |
| 动态（焦点、进度） | `activeContext.md`、`progress.md` | 频繁 |

Cline Memory Bank 的关键洞察：**动态状态必须单独成文件**，否则 Claude 找不到"我现在在哪"。

## 3. 跨仓库优先，单项目兼容

很多 memory bank 工具都是"一个 repo 一份 memory"。但现实是：

- 后端依赖 proto 共享库
- 前端依赖 design-tokens 仓库
- 多个微服务共享脚手架

**vault 首页的依赖图是核心价值**。Claude 看一眼就知道"改这个会影响那个"。

## 4. 纯 markdown，零依赖

不用向量数据库、不用知识图谱、不用插件。**理由**：

- markdown 任何工具都能读（Claude / Codex / 你自己 / 同事）
- 离开 Claude 也能用（手机 Obsidian 翻）
- 永远不会"加载失败"
- 迁移成本零（复制走就行）

## 5. 语言无关

模板里没有 Go / Python / JS 任何特定假设。`git diff` 解析也用通用方式。

理由：用户的多个仓库可能技术栈不同（前端 TS、后端 Go、proto、脚本 Python），强制单一语言假设会立刻崩。

## 6. 可选自动化

- **基础功能**（vault 结构 + 模板）：必需
- **commands**（`/sync-docs` 等）：强烈推荐
- **hooks**（Stop 自动写日志）：可选

不开启自动化也能用。开启后体验更顺。

## 7. 不替代 git，但和 git 协同

Git 记**做了什么**（commit message + diff），文档记**为什么 + 当前状态**。

`/commit` 命令把两件事绑在一起：commit 后自动用本次 diff 更新对应项目的 `progress.md` 或 `activeContext.md`。

## 8. Codex 兼容是 bonus，不是核心

主目标是 Claude Code 用户。Codex 用户能用 vault 结构 + AGENTS.md，但 commands / hooks 没有。

不为了 Codex 兼容牺牲 Claude Code 体验。

## 9. 单人优先，团队后续考虑

现在不做团队协作。理由：

- vault 是高度个人化的（每个人对"焦点"的定义不同）
- 团队协作可以用 Obsidian Sync 或 git，不需要 skill 介入
- 强行做团队会损害单人体验

## 10. 不做 second brain

只管开发项目跟踪。**理由**：

- second brain 是 Tiago Forte 的领地，已经很卷
- 通用知识管理是无底洞，定位不清晰
- 用户可以**同时**用其他 second brain skill，互不冲突
